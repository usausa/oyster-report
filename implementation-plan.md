# OysterReport 実装計画

## 1. このドキュメントの目的

このドキュメントは、`library-spec.md` に基づいて OysterReport を段階的に実装していくための作業指示書である。

- 詳細仕様、データモデル定義、未サポート事項、設計判断理由は `library-spec.md` を正とする
- このドキュメントは「どの順番で」「どの単位で」「どのテストを先に追加しながら」進めるかを定義する
- 各ステップは、小さく完了判定できる単位で進める
- 各ステップでユニットテストを先に追加し、最小実装、整理、回帰確認の順で進める

## 3. 実装の進め方

各ステップは以下の流れで進める。

1. そのステップで必要なテストケースを追加する
2. テストが失敗することを確認する
3. 最小限の実装でテストを通す
4. 必要ならリファクタリングする
5. 追加したテストと既存テストをすべて通す
6. `library-spec.md` と差異がないか確認する

各ステップで守ること:

- 1回の変更範囲を広げすぎない
- 公開 API を先に増やしすぎない
- internal 設計は必要になるまで露出しない
- 中間データ構造に PDF 描画専用の計算結果を混ぜない
- レイアウト変更ではなく、帳票生成に必要な制御メソッド中心で進める

## 4. 成果物方針

実装作業で最低限そろえる成果物は以下とする。

- ライブラリ本体プロジェクト
- ユニットテストプロジェクト
- 必要に応じたテスト用 Excel テンプレート
- 必要に応じた PDF 比較用またはダンプ比較用の期待値データ
- 実装完了後に見直した `README.md`

テストプロジェクトは少なくとも以下の2系統を持つ想定とする。

- 純粋なモデル、変換、計算ロジックを検証するユニットテスト
- Excel 入力や PDF 出力を含む統合寄りのテスト

## 6. ステップ横断の確認項目

各ステップの最後に、必ず以下を確認する。

- `library-spec.md` と公開 API がずれていないか
- 中間データ構造に PDF 描画専用情報を混ぜていないか
- hidden 行、hidden 列の扱いが一貫しているか
- ヘッダ、フッタの考慮が落ちていないか
- 要検討項目を、実装で黙って固定化していないか
- 未対応機能が診断なしで無視されていないか

## 7. 推奨テスト資産

用意しておくとよいテスト資産は以下。

- 最小テンプレート

- hidden 行、hidden 列ありテンプレート

## 8. 実装時の優先順位

優先順位は以下の通りとする。

1. 中間データ構造の確立
2. 読み込み
3. 帳票編集
4. internal レンダリング計画
5. PDF 描画
6. 診断とダンプ
7. レイアウト忠実度の強化

## 9. この計画書の使い方

実装開始時は、毎回以下を行う。

1. 今回着手する Step を1つ選ぶ
2. 該当 Step の参照先として `library-spec.md` の該当章を確認する
3. 先に追加するテストを書き、失敗を確認する
4. 最小実装で通す
5. 次の Step へ進む前に、公開 API と internal 境界を見直す

このドキュメントは実装順を管理するためのものであり、仕様の最終判断は常に `library-spec.md` を参照する。
